## Adding to this list

Please ensure your pull request adheres to the following guidelines:

- **Search existing issues and pull requests** before making a new one, as yours may be a duplicate.
- **Only add items that are directly related to stealth tradecraft** – tools, techniques, research, or resources that help operators move, communicate, or persist without creating noise in telemetry, logs, or detection stacks.
- **Prefer Free/Open Source Software** – while not strictly required, closed‑source or commercial tools should be clearly marked and have a strong justification.
- **Make sure the item is awesome and useful** – it should be actively maintained, well‑documented, or a known reference in the field.
- **Make an individual pull request for each suggestion** – this makes reviewing easier.
- **Use [title‑casing](http://titlecapitalization.com) (AP style)** for item names.
- **Use the following format:** `- [Item Name](https://link.to/tool) – Short description of what it does and why it's relevant to silent security.`
- **Link additions should be added to the bottom of the relevant category.** If a new category makes sense, suggest it in the PR.
- **New categories or improvements to the existing categorisation are welcome** – but please open an issue first to discuss major structural changes.
- **Check your spelling and grammar.**
- **Make sure your text editor is set to remove trailing whitespace.**
- **The pull request and commit should have a useful title** (e.g., `Add AwesomeTool for log tampering`).
- **The body of your commit message should contain a link to the repository** or the resource you're adding.

Thank you for helping make this list the definitive resource for noise‑free tradecraft!

## Adding something to an awesome list

If you have something awesome to contribute to this awesome list, this is how you do it.

You'll need a [GitHub account](https://github.com/join)!

1. Access the awesome list's GitHub page. For example: https://github.com/MOUKA-513/awesome-silent-security
2. Click on the `README.md` file.  
   ![Step 2 Click on Readme.md](https://cloud.githubusercontent.com/assets/170270/9402920/53a7e3ea-480c-11e5-9d81-aecf64be55eb.png)
3. Now click on the edit icon (the pencil).  
   ![Step 3 - Click on Edit](https://cloud.githubusercontent.com/assets/170270/9402927/6506af22-480c-11e5-8c18-7ea823530099.png)
4. You can start editing the text of the file in the in‑browser editor. Make sure you follow the guidelines above. You can use [GitHub Flavored Markdown](https://help.github.com/articles/github-flavored-markdown/).  
   ![Step 4 - Edit the file](https://cloud.githubusercontent.com/assets/170270/9402932/7301c3a0-480c-11e5-81f5-7e343b71674f.png)
5. Write a clear commit message explaining why you're proposing the changes, and then click on **"Propose file change"**.  
   ![Step 5 - Propose Changes](https://cloud.githubusercontent.com/assets/170270/9402937/7dd0652a-480c-11e5-9138-bd14244593d5.png)
6. Submit the [pull request](https://help.github.com/articles/using-pull-requests/)!

## Updating your Pull Request

Sometimes, a maintainer of this list will ask you to edit your Pull Request before it is included. This is normally due to spelling errors, broken links, or because your PR didn't match the contribution guidelines.

- Make the requested changes in your local branch.
- Push the changes to your fork – the pull request will update automatically.
- Leave a comment on the pull request once you've made the changes.
